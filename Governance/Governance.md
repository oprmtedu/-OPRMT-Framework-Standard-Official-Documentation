## 1. Purpose

This document defines the governance structure for the **OPRMT™ Framework Standard Official Documentation** repository.

The purpose of this governance file is to establish clear authority, review standards, contribution control, versioning rules, documentation ownership, and repository management expectations for the OPRMT™ Framework.

The OPRMT™ Framework is not treated as an informal prompt collection. It is treated as a structured framework documentation system requiring consistent terminology, controlled updates, and formal review.

---

## 2. Governance Objective

The objective of OPRMT™ governance is to protect the accuracy, consistency, ownership, and long-term integrity of the OPRMT™ Framework.

Governance ensures that:

- The framework remains clearly defined
- Official terminology remains consistent
- Documentation changes are reviewed before acceptance
- Contributors understand the review process
- Version history remains traceable
- Repository structure remains organized
- Trademark and intellectual property rights are protected
- Public-facing documentation maintains professional quality
- Framework changes are not made casually or without review

## 6. Governance Principles

The OPRMT™ governance model is based on the following principles.

| Principle | Description |
|---|---|
| Canonical Accuracy | Official definitions must remain consistent and controlled |
| Documentation Integrity | Repository content must remain structured, professional, and usable |
| Version Control | Changes must be traceable through version history |
| Review Before Acceptance | Contributions must be reviewed before becoming official |
| IP Protection | Ownership, trademark, and commercial rights must be protected |
| Public Clarity | Documentation should be understandable without misrepresenting authority |
| Compliance Awareness | Claims, examples, and educational materials must avoid unsupported statements |
| Maintainability | Repository files should remain easy to update, audit, and reference |
| Framework Alignment | All accepted content must align with Objective, Parameters, Results, Method, and Tools |

## 7. Governance Roles

The repository may include the following governance roles.

| Role | Description |
|---|---|
| Owner | Final authority over repository governance, framework meaning, releases, and licensing |
| Maintainer | Authorized person responsible for reviewing, editing, and merging accepted changes |
| Reviewer | Person assigned to evaluate submissions for quality, accuracy, or compliance |
| Contributor | Person submitting issues, pull requests, documents, prompts, or suggestions |
| User | Person reading, referencing, or using the repository materials |
| Community Member | Person participating in discussion, feedback, or educational use |

The same person may hold multiple roles.

Canonical documentation means the official documentation accepted and maintained within the authorized OPRMT™ Framework Standard Official Documentation repository.

Canonical documents may include:

- Framework overview
- Objective specification
- Parameters specification
- Results specification
- Method specification
- Tools specification
- Prompt templates

- Versioning policy
- Style guide

## 13. Framework Change Control

Changes to the OPRMT™ Framework must be reviewed before becoming official.

Framework-level changes include:

- Changing the meaning of Objective
- Changing the meaning of Parameters
- Changing the meaning of Results
- Changing the meaning of Method
- Changing the meaning of Tools
- Adding new official framework components
- Removing existing framework components
- Changing canonical definitions
- Changing core terminology
- Changing versioning policy
- Changing governance authority
- Changing contribution rules

Framework-level changes require elevated review because they may affect the entire documentation system.

---

## 14. Documentation Change Control

Documentation changes may be classified by impact level.

| Change Type | Description | Review Level |
|---|---|---|
| Typo Correction | Spelling, grammar, punctuation, formatting | Low |
| Link Repair | Fixing broken or outdated links | Low |
| Clarification | Improving explanation without changing meaning | Medium |
| New Section | Adding new official content | Medium |
| Structural Change | Reorganizing file or repository structure | Medium / High |
| Framework Definition Change | Modifying canonical meaning | High |
| Governance Change | Modifying review, ownership, or approval rules | High |
| License or IP Change | Modifying usage rights or legal terms | High |

Low-impact changes may be reviewed quickly. High-impact changes require owner or authorized maintainer approval.

## 15. Contribution Governance

All contributions must follow the repository contribution process.

Contribution types may include:

- Issues
- Pull requests
- Change requests
- Document requests
- Prompt submissions
- Example submissions
- Correction suggestions
- Governance proposals

A contribution may be accepted only if it:

- Fits repository scope
- Aligns with OPRMT™ terminology
- Preserves framework meaning
- Uses professional documentation standards
- Avoids unsupported claims
- Does not violate intellectual property rules
- Does not introduce security risk
- Supports repository maintainability

Contributions may be rejected if they are incomplete, off-topic, legally risky, low-quality, duplicative, or inconsistent with the OPRMT™ Framework.

---

## 16. Review Standards

Submissions should be reviewed against the following standards.

| Review Area | Review Question |
|---|---|
| Scope | Does the contribution belong in this repository? |
| Accuracy | Is the content technically and conceptually accurate? |
| Framework Alignment | Does the content align with OPRMT™? |
| Terminology | Does the content use approved terms consistently? |
| Documentation Quality | Is the content clear, structured, and professional? |
| Legal/IP | Does the submission avoid unauthorized third-party content? |
| Claims Control | Does the content avoid unsupported legal, financial, accreditation, or adoption claims? |
| Versioning | Does the change require a version update? |
| Maintainability | Can this content be maintained long term? |
| Public Readiness | Is the content suitable for public repository use? |

---

## 17. Approval Requirements

A contribution may be approved only after it satisfies all applicable review standards.

At minimum, approved contributions should meet these requirements:

- Clear purpose
- Correct file placement
- Proper Markdown formatting
- Consistent terminology
- No unsupported claims
- No confidential information
- No private credentials
- No unauthorized copyrighted material
- No conflict with official framework definitions
- No conflict with repository license
- Maintainer or owner approval

High-impact contributions may require additional review before approval.

---

## 18. Rejection Criteria

A contribution may be rejected if it:

- Conflicts with OPRMT™ definitions
- Misrepresents the framework
- Attempts to redefine official terminology without approval
- Includes unsupported claims
- Includes confidential information
- Includes private credentials
- Violates licensing terms
- Copies unauthorized third-party material
- Is off-topic
- Is promotional spam
- Is poorly structured
- Duplicates existing documentation
- Creates unnecessary maintenance burden
- Introduces legal or reputational risk
- Attempts to transfer or dilute ownership rights

Maintainers may reject contributions without obligation to merge or preserve them.

## 19. Versioning Governance

The repository should use controlled versioning to track changes.

Recommended versioning model:

| Version Type | Use Case |
|---|---|
| Patch | Typo fixes, link fixes, minor formatting corrections |
| Minor | New examples, clarifications, small documentation additions |
| Major | Framework definition changes, governance changes, structural standard changes |

Version updates should be reflected where applicable in:

- Document metadata
- Revision history
- `CHANGELOG.md`
- Release notes
- Repository tags
- Specification references

Versioning should preserve historical traceability.

Official releases may be created when the repository reaches a stable documentation state.

A release may include:

- Version number
- Release date
- Summary of changes
- Files affected
- Breaking changes, if any
- Deprecated files, if any

OPRMT™ terminology must remain consistent.

Official framework terms:

| Term | Required Usage |
|---|---|
| OPRMT™ | Use with trademark symbol where appropriate in formal documentation |
| Objective | Defines the goal |
| Parameters | Defines rules, context, boundaries, and constraints |

| DOAS Scoring | Use only in alignment with approved scoring documentation |

## 25. Claims Governance

Public documentation must avoid unsupported claims.

Do not include unsupported claims such as:

- Guaranteed financial results
- Guaranteed business success
- Legal compliance guarantees
- Government recognition
- University accreditation
- Industry-wide adoption
- Medical, legal, or financial advice
- Unverified testimonials
- Claims of official third-party endorsement
- Certification recognition beyond what is actually offered

Acceptable language should be factual, controlled, and specific.

Example of acceptable wording:

~~~text
The OPRMT™ Certified Prompt Engineer course is an OPRMT™ educational certification program.
~~~

Example of risky wording:

~~~text
This certification is recognized by all companies and guarantees professional success.
~~~

---

## 26. Certification Governance

Any certification-related documentation must be controlled carefully.

Certification documents should clearly state:

- Who issues the certification
- What completion means
- What learners are assessed on
- Whether the certification is accredited or non-accredited
- What rights the certification does and does not grant
- What conduct standards apply
- What usage rules apply to certificate holders

Certification documentation must not claim accreditation, legal recognition, employment guarantees, or universal industry acceptance unless independently verified and approved.

---

## 27. Prompt Compliance Engineering Governance

Prompt Compliance Engineering references should be used consistently.

PCE-related content should:

- Explain compliance as a structured prompt review discipline
- Avoid claiming formal legal compliance unless legally reviewed
- Distinguish between prompt quality review and legal/regulatory compliance
- Reference OPRMT™ as a structured framework where appropriate
- Use clear review criteria
- Avoid unsupported professional licensing claims

PCE documentation should be placed in the appropriate repository folder and reviewed before publication.

---

## 28. DOAS Scoring Governance

DOAS scoring references should be used only when aligned with approved scoring documentation.

DOAS-related content should define:

- What is being scored
- What criteria are used
- How scores are interpreted
- What the score does not guarantee
- Whether scoring is manual, automated, or hybrid
- Whether scoring is advisory, educational, or operational

DOAS should not be presented as a legal certification, regulatory approval, or guaranteed performance measure unless separately validated and approved.

---

## 29. Security Governance

The repository must not include private or sensitive credentials.

Do not commit:

- API keys
- Passwords
- Access tokens
- Private keys
- Personal identity documents
- Confidential business documents
- Private customer data
- Secret URLs
- Internal-only files
- Payment credentials
- Private Google Drive access credentials

Security-related issues should be reported through the process defined in `SECURITY.md`.

If a secret is accidentally committed, it should be revoked immediately and removed from the repository history if necessary.

---

## 30. Licensing Governance

Repository content must comply with the official license.

Licensing governance should define:

- Permitted use
- Prohibited use
- Attribution requirements
- Commercial use restrictions
- Redistribution rules
- Modification rules
- Contribution ownership terms
- Disclaimer of warranties
- Enforcement rights

No contributor may override the repository license through a pull request, issue, example file, or separate statement unless approved by the repository owner.

---

## 31. Trademark Governance

OPRMT™ should be used consistently and respectfully.

Trademark governance includes:

- Do not imply ownership of OPRMT™ unless authorized
- Do not use OPRMT™ to market unrelated products without permission
- Do not create confusingly similar framework names
- Do not present unofficial forks as official
- Do not imply certification authority without approval
- Do not remove ownership or official notices from documentation

The trademark symbol should be used in formal title references where appropriate.

---

## 32. Fork Governance

Users may fork a repository only within the rights granted by the repository license.

Forks must not:

- Claim to be the official OPRMT™ repository
- Remove required ownership notices
- Misrepresent governance authority
- Sell restricted materials without permission
- Issue unauthorized certifications
- Publish altered definitions as official OPRMT™ definitions
- Create confusion about canonical documentation

Any official fork policy should be consistent with the repository license.

---

## 33. Issue Governance

Issues should be used for:

- Bug reports
- Documentation errors
- Link problems
- Clarification requests
- Improvement suggestions
- Change requests
- Questions related to repository use

Issues should not be used for:

- Spam
- Harassment
- Promotional posting
- Unsupported legal claims
- Unauthorized advertising
- Unrelated technical support
- Demands for immediate acceptance
- Attempts to override governance authority

Maintainers may close issues that are off-topic, duplicative, abusive, or inconsistent with repository scope.

---

## 34. Pull Request Governance

Pull requests should be reviewed before merging.

A pull request should include:

- Clear title
- Summary of changes
- Files changed
- Reason for the change
- Version impact, if applicable
- Confirmation of original or authorized content
- Any relevant issue references

Pull requests may be:

- Approved
- Approved with edits
- Returned for changes
- Deferred
- Rejected
- Closed without merge

No pull request is automatically accepted.

---

## 35. Change Request Governance

A change request should be used when a proposed update may affect framework meaning, repository structure, governance, or documentation standards.

Change requests should include:

- Current state
- Proposed change
- Reason for change
- Files affected
- Version impact
- Risks
- Benefits
- Review category
- Approval status

Framework-level change requests require owner or authorized maintainer approval.

---

## 36. Documentation Quality Governance

Documentation should meet a professional public-facing standard.

Official documentation should be:

- Clear
- Structured
- Complete
- Accurate
- Maintainable
- Searchable
- Reusable
- Versioned
- Professionally written
- Free from unsupported claims
- Aligned with OPRMT™ terminology

Documentation should not be vague, overly promotional, inconsistent, or written in a casual style unsuitable for repository use.

---

## 37. Deprecation Governance

Documents or sections may be deprecated when they become outdated, replaced, or no longer aligned with the current framework.

Deprecated content should be marked clearly.

Recommended deprecation notice:

~~~text
Deprecated: This document has been replaced by [new-document.md]. It is retained for historical reference only and should not be used as the current standard.
~~~

Deprecated content should not be deleted immediately if it is needed for version history, migration, or reference.

---

## 38. Archival Governance

Archived materials should be separated from active documentation.

Archived content may include:

- Older framework drafts
- Superseded specifications
- Deprecated examples
- Historical release notes
- Experimental documents
- Prior templates

Archived materials should not be presented as current canonical documentation.

Suggested folder:

~~~text
/archive/
~~~

- Explanation of rejection
- Requested changes
- Reference to governance rules
- Deferred review
- Final decision

Final authority remains with the repository owner or authorized maintainers.

## 42. Governance Update Process

Governance updates should follow a controlled process.

Recommended process:

1. Identify the governance issue.
2. Draft the proposed update.
3. Review the impact on existing repository rules.
4. Confirm alignment with ownership and licensing requirements.
5. Update the document version.
6. Record the change in revision history.
7. Update related documents if needed.
8. Publish the approved governance update.

Governance changes should not be made casually because they affect repository authority and contribution control.
